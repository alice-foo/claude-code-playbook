# Swift + SwiftUI + Xcode アプリ開発向けの初期化

## 行動指針

以下の設計原則に従って行動してください：

- **計画先行の開発**: コードを書く前に必ず計画を立てる
- **対話式の要件定義**: ユーザへのインタビューで要件を明確にする
- **簡潔な設定ファイル**: プロジェクトのルール・コマンド・構造を CLAUDE.md に集約する
- **段階的な実装**: 一度にすべてを作らず、フェーズごとに進める
- **毎回のビルド検証**: 各フェーズでビルド・テストが通ることを確認してから次に進む

## 技術スタック

- **プラットフォーム**: インタビューで確定（iOS / macOS / watchOS / visionOS）
- **言語**: Swift（最新安定版）
- **UI**: SwiftUI
- **IDE**: Xcode
- **アーキテクチャ**: MVVM（インタビューで変更可）

## 進め方

### フェーズ 0: 環境確認 → インタビュー（今ここ）

Plan Mode で進めます。**コードは書かないでください。**

まず `xcodebuild -version` を実行し、Xcode が利用可能か確認してください。
コマンドが見つからない場合は、その旨をユーザに伝えて中断してください。

確認後、AskUserQuestion ツールを使って私にインタビューし、以下を明確にしてください：

1. アプリの目的・解決したい課題
2. 主要機能（3〜5 個に絞る）
3. 対象プラットフォーム — iPhone（iOS）/ Mac（macOS）/ 両方。対応させたい最低 OS バージョン
4. データの保存先 — アプリのデータをどこに保存するか
   - **なし** — データを保存しない
   - **UserDefaults** — 少量の設定値のみ
   - **SwiftData** — Apple 推奨の新しいデータ保存の仕組み（iOS 17 以降）
   - **CoreData** — Apple 従来のデータ保存の仕組み（古い OS もサポートしたい場合）
5. 外部サービスとの連携 — 他社の Web サービス（地図 API、決済 API など）を呼び出すか、ログイン機能は必要か
6. テスト — 自動テスト（コードが正しく動くかを自動で検証する仕組み）を書くか。書く場合、画面操作のテストも含めるか
7. 自動化 — コードを GitHub に push したとき、ビルドやテストを自動実行する仕組み（CI/CD）が必要か
8. バージョン番号の方針 — アプリのバージョンをどう付けるか
   - 開始番号: 0.1.0 から始めるか、1.0.0 から始めるか
   - 形式: メジャー.マイナー.パッチ（例: 1.2.3）にするか、ビルド日時を含めるか（例: 1.2.3 (20251231.2359)）
   - 特にこだわりがなければ 0.1.0 (yyyyMMdd.HHmm) のように、開始番号は 0.1.0 を、ビルド日時は含める形式を提案する
9. 特殊な機能 — アプリ内課金、プッシュ通知、ウィジェットなど、気になる機能があれば

質問は一度に全部ではなく、回答に応じて深掘りしてください。

### フェーズ 1: 設計ドキュメント生成

インタビュー完了後、以下を生成してください：

- **CLAUDE.md**（200 行以内）
  - プロジェクト概要
  - ビルドコマンド: `xcodebuild -scheme <Scheme> -destination 'platform=...'`
  - テストコマンド: `xcodebuild test -scheme <Scheme> -destination '...'`
  - コーディング規約（async/await 優先、guard let 使用、force unwrap 禁止）
  - ディレクトリ構成の説明
- **プロジェクト構造案**（ツリー形式）
- **フェーズ分けされた実装計画**（各フェーズはコンテキストの 50% 以内で完了する粒度）

### フェーズ 2 以降: 実装サイクル

私が計画を承認したら、フェーズごとに以下を繰り返します：

1. 実装
2. `xcodebuild` でビルド確認
3. テスト実行
4. git commit（フェーズ単位）

ビルドが通らない場合は次のフェーズに進まず、その場で修正してください。
